## 决策1: 沙箱方案选型

### 问题背景

**需求**: 安全执行LLM生成的Manim代码  
**挑战**: 
- LLM生成的代码不可信
- 需要隔离执行环境
- 需要资源限制
- 需要考虑性能和成本

### 方案对比

#### 方案A: 直接Docker

**优点:**
- 简单直接，社区成熟
- 文档完善，易于使用
- 性能开销适中

**缺点:**
- 需要手动管理容器生命周期
- 缺少高级API
- 需要自己实现文件操作、代码执行等功能

#### 方案B: OpenSandbox（选择）✅

**优点:**
- 阿里开源，专为AI应用设计
- 统一的SDK，开发简单
- 支持多种运行时（Docker、Firecracker、gVisor）
- 内置代码解释器、文件操作
- 中文文档完善

**缺点:**
- 相对较新（2026年3月发布）
- 社区生态还在发展中

#### 方案C: Firecracker

**优点:**
- 性能最优（启动快2-4倍）
- 内存占用最低（减少50%）
- 硬件级隔离

**缺点:**
- 配置复杂
- 需要准备rootfs、kernel
- 调试困难

#### 方案D: Python限制执行

**优点:**
- 最轻量，无额外依赖
- 性能开销最低

**缺点:**
- 安全性不足
- 不适合生产环境

### 决策结果

**选择**: OpenSandbox + Docker运行时

**理由:**
1. **开发效率**: SDK封装完善，减少开发时间
2. **灵活性**: 未来可无缝切换到Firecracker
3. **安全性**: 容器级隔离满足需求
4. **生态**: 阿里开源，中文文档完善
5. **成本**: 开源免费，运维成本低

**权衡:**
- 性能略低于Firecracker，但在可接受范围
- OpenSandbox较新，但背靠阿里，稳定性有保障

## 决策2: 动画技术栈选择

### 问题背景

**需求**: 生成数学教学动画  
**要求**:
- 支持数学公式渲染
- 支持坐标系、函数图像
- 动画质量高
- 适合程序化生成

### 方案对比

#### 方案A: Manim（选择）✅

**优点:**
- 专为数学设计（3Blue1Brown开发）
- Python原生，与项目技术栈匹配
- 支持数学公式、坐标系、函数图像
- 代码可程序化生成
- 社区活跃，文档完善

**缺点:**
- 学习曲线较陡
- 渲染时间较长（10-30秒）

#### 方案B: D3.js

**优点:**
- 前端可视化库
- 交互性强
- 渲染快

**缺点:**
- 不是专门为数学设计
- 需要前端实现
- 公式渲染不如Manim

#### 方案C: Matplotlib动画

**优点:**
- Python原生
- 简单易用
- 性能好

**缺点:**
- 动画效果一般
- 不如Manim专业
- 公式渲染能力弱

#### 方案D: Canvas/WebGL

**优点:**
- 性能最优
- 实时渲染

**缺点:**
- 需要大量前端开发
- 数学功能需要自己实现
- 开发成本高

### 决策结果

**选择**: Manim

**理由:**
1. **专业性**: 专为数学教学设计
2. **技术栈**: Python原生，与后端完美集成
3. **功能性**: 支持数学公式、坐标系、函数图像
4. **质量**: 动画质量高，适合教学
5. **可维护性**: 代码可程序化生成，易于维护

**权衡:**
- 渲染时间较长，但通过缓存可优化
- 学习曲线陡，但通过模板可降低

## 决策3: 部署架构设计

### 问题背景

**需求**: 部署动画生成服务  
**场景**: 开发环境、小规模生产、大规模生产

### 方案对比

#### 方案A: 单一部署方案

**优点:**
- 配置简单

**缺点:**
- 无法适应不同规模
- 性能和成本不可优化

#### 方案B: 分阶段部署（选择）✅

**阶段1: 本地开发**
```
Docker Desktop + OpenSandbox Server
```

**阶段2: 小规模生产**（≤100并发）
```
单服务器 + Docker + OpenSandbox
```

**阶段3: 大规模生产**（>100并发）
```
Kubernetes + OpenSandbox + Firecracker
```

**优点:**
- 按需扩展
- 成本可控
- 灵活调整

**缺点:**
- 需要维护多套配置

### 决策结果

**选择**: 分阶段部署

**理由:**
1. **成本优化**: 按需使用资源
2. **灵活扩展**: 根据用户量调整
3. **风险可控**: 逐步升级
4. **技术演进**: 从简单到复杂

**权衡:**
- 需要维护多套配置，但通过OpenSandbox的运行时切换可以简化

## 决策4: 代码生成策略

### 问题背景

**需求**: 生成Manim代码  
**选项**: 模板 vs 完全LLM生成

### 方案对比

#### 方案A: 纯模板

**优点:**
- 速度快
- 质量稳定
- 无LLM成本

**缺点:**
- 灵活性差
- 难以覆盖所有场景

#### 方案B: 纯LLM生成

**优点:**
- 最灵活
- 可处理复杂场景

**缺点:**
- 质量不稳定
- 成本高
- 速度慢

#### 方案C: 混合策略（选择）✅

```python
if animation_type in TEMPLATES:
    # 使用模板
    code = render_template(animation_type, params)
else:
    # 使用LLM生成
    code = generate_by_llm(animation_type, params)
```

**优点:**
- 兼顾速度和灵活性
- 降低LLM成本
- 保证常见场景质量

**缺点:**
- 需要维护模板库

### 决策结果

**选择**: 混合策略

**理由:**
1. **成本优化**: 常见场景使用模板，无LLM成本
2. **质量保证**: 模板质量可控
3. **灵活性**: LLM处理复杂场景
4. **性能**: 模板生成快，LLM生成慢

**权衡:**
- 需要维护模板库，但收益大于成本

## 决策5: 缓存策略

### 问题背景

**需求**: 减少重复计算，提升响应速度

### 方案对比

#### 方案A: 无缓存

**优点:**
- 实现简单
- 数据最新

**缺点:**
- 性能差
- 成本高

#### 方案B: 基于参数的缓存（选择）✅

```python
cache_key = md5(f"{animation_type}:{params}")
if cache.exists(cache_key):
    return cache.get(cache_key)
```

**优点:**
- 相同参数自动缓存
- 大幅减少重复计算
- 实现简单

**缺点:**
- 需要管理缓存空间

#### 方案C: Redis缓存

**优点:**
- 分布式缓存
- 持久化

**缺点:**
- 增加依赖
- 配置复杂

### 决策结果

**选择**: 基于参数的文件缓存

**理由:**
1. **简单**: 无需额外依赖
2. **有效**: 大幅减少重复计算
3. **可控**: 可手动清理缓存
4. **成本低**: 使用本地存储

**后续优化**:
- MVP阶段使用文件缓存
- 生产环境可切换到Redis
