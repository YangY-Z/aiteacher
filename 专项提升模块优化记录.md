# 专项提升模块优化记录

## 本次改动（2026-03-28）

### 问题 1：诊断和澄清问答过多
**现象：** 追问轮次没有上限控制，可能导致用户体验不佳

**解决方案：**
- 新增 `max_clarification_rounds` 配置项，默认值 5，可调范围 1-5
- 后端模型层：`ImprovementSession` 增加 `max_clarification_rounds` 字段
- 后端 API 层：`ScoreUploadRequest` schema 增加该字段
- 后端服务层：`submit_clarification_answer` 方法使用 `session.max_clarification_rounds` 判断是否达到上限
- 前端表单：新增"最多追问次数"输入框，默认值 5
- 前端展示：澄清卡片右上角显示 `已追问次数/上限`

**涉及文件：**
- `ai-teacher-backend/app/models/improvement.py`
- `ai-teacher-backend/app/schemas/improvement.py`
- `ai-teacher-backend/app/services/improvement_service.py`
- `ai-teacher-backend/app/api/improvement.py`
- `ai-teacher-frontend/src/types/index.ts`
- `ai-teacher-frontend/src/pages/Improvement.tsx`

### 问题 2：分步学习点击"开始本步"没有反应
**现象：** 点击按钮后没有明显反馈，用户不知道是否成功

**解决方案：**
- 后端服务层：`start_plan_step` 方法改为真实生成教学内容
  - 优先调用 LLM 生成结构化讲解（使用 `TEACHING_PROMPT`）
  - LLM 失败时使用降级内容（包含引入、定义、示例、思考题、总结）
  - 返回完整的教学内容结构（content + whiteboard）
- 前端展示：
  - 点击后显示成功提示："已加载第 X 步学习内容"
  - 在方案下方展示学习内容卡片，包含：
    - 引入
    - 定义/要点
    - 示例
    - 思考题
    - 总结
    - 板书公式（如有）
    - 建议图示（如有）

**涉及文件：**
- `ai-teacher-backend/app/services/improvement_service.py`
- `ai-teacher-frontend/src/pages/Improvement.tsx`

### 其他修复
- 修复后端 `_build_session_response` 函数中变量名冲突导致返回错误的 bug
- 前端 TypeScript 类型收窄，避免 `unknown` 类型编译错误

## 后续可优化的点

### 1. 分步学习交互增强（优先级：高）
**当前状态：** 学习内容以静态卡片形式展示，用户只能阅读后手动点击"标记完成"

**优化方向：**
- 参考 `/learn` 页面的交互模式，增加：
  - 学生可以在学习内容下方提问或回答思考题
  - AI 根据学生回答给出反馈
  - 支持白板公式的 KaTeX 渲染
  - 支持图示的可视化展示
- 增加学习进度追踪：
  - 记录每一步的学习时长
  - 记录学生的互动次数和质量
  - 根据互动情况判断是否真正掌握，而不是简单点击"完成"

**实现建议：**
- 复用 `ai-teacher-frontend/src/components/chat/` 组件
- 复用 `ai-teacher-frontend/src/components/whiteboard/` 组件
- 在 `improvement_service.py` 中增加 `process_step_message` 方法处理学生互动
- 前端增加 `ImprovementLearningPanel` 组件，类似 `Learning.tsx` 的交互模式

### 2. 诊断准确性提升（优先级：中）
**当前状态：**
- LLM 诊断 + 规则降级
- 澄清问题由 LLM 生成或使用预设问题

**优化方向：**
- 增加历史错题分析：
  - 如果学生之前在该课程做过评估，分析历史错题类型
  - 结合历史数据和当前成绩输入，提高诊断准确性
- 优化澄清问题生成：
  - 根据候选知识点的特征生成更有针对性的问题
  - 避免泛泛的"你哪里不会"，改为具体场景题
- 增加置信度阈值：
  - 当诊断置信度低于某个阈值时，建议学生做一套快速诊断题
  - 根据诊断题结果精确定位薄弱点

### 3. 学习方案个性化（优先级：中）
**当前状态：**
- LLM 生成 + 规则降级
- 方案步骤相对固定（前置补缺 → 核心讲解 → 练习 → 小测）

**优化方向：**
- 根据学生自评基础调整方案深度：
  - 基础薄弱：增加前置知识补缺步骤，降低难度
  - 基础较好：减少基础讲解，增加拓展练习
- 根据可用时间动态调整：
  - 时间充裕：增加练习题数量和变式训练
  - 时间紧张：聚焦核心概念和典型题型
- 增加学习路径选择：
  - 理论优先：先系统讲解再练习
  - 实践优先：先做题发现问题再针对性讲解

### 4. 最终小测优化（优先级：低）
**当前状态：**
- 固定从题库抽取 4 道题
- 简单评分（正确率 >= 60% 为通过）

**优化方向：**
- 自适应出题：
  - 根据学生在分步学习中的表现调整题目难度
  - 如果学生某个子知识点掌握不好，增加该部分题目
- 增加错题讲解：
  - 小测失败后，针对错题生成详细讲解
  - 提供相似题目供学生巩固
- 增加重测机制：
  - 允许学生在学习后重新测试
  - 记录多次测试的进步曲线

### 5. 数据持久化和分析（优先级：低）
**当前状态：**
- 专项提升会话保存在 JSON 文件
- 没有跨会话的数据分析

**优化方向：**
- 增加专项提升历史记录页面：
  - 展示学生历次专项提升的目标知识点
  - 展示每次的诊断结果和最终测试成绩
  - 分析学生的薄弱知识点变化趋势
- 增加学习效果评估：
  - 对比专项提升前后在该知识点的掌握程度
  - 统计专项提升的平均完成时间和通过率
- 为教师提供数据看板：
  - 班级整体薄弱知识点分布
  - 学生专项提升使用频率和效果

## 技术债务

### 1. 前端类型定义不够严格
- `stepContent` 使用 `Record<string, unknown>`，导致需要大量类型断言
- 建议定义明确的 `StepContentResponse` 接口

### 2. 后端 LLM 调用缺少重试机制
- 当前 LLM 失败直接降级，没有重试
- 建议增加指数退避重试策略

### 3. 前端状态管理可以优化
- `stepContent` 和 `session` 分开管理，可能导致状态不一致
- 建议将 `stepContent` 合并到 `session` 中统一管理
